The **CAP theorem** (pronounced "Cap") is a fundamental concept in computer science that describes the trade-offs inherent in distributed systems. It was proposed by Eric Brewer in 2000 and later formalized by the open-source community in 2001.

Here is an explanation of the theorem and its practical implications for distributed databases.

### 1. The Definitions: C, A, and P

The theorem states that a distributed system can only satisfy two of the following three properties simultaneously:

*   **C - Consistency:** Every read receives the most recent write or an error.
    *   *What it means:* All nodes in the cluster see the same data at the same time. If you write data to Node A, you can read it from Node B immediately. It is a "single source of truth."
*   **A - Availability:** Every request receives a response (success or error) without failure of the node.
    *   *What it means:* The system remains operational. Even if a node is down or a network failure occurs, the system still responds to requests. It prioritizes uptime over data accuracy.
*   **P - Partition Tolerance:** The system continues to operate despite an arbitrary loss of a portion of the system's inter-node communication (network failure).
    *   *What it means:* The system handles "split-brain" scenarios. If the network breaks and nodes cannot talk to each other, the system does not crash.

### 2. The Logic: Why you can't have all three

The theorem implies that **Consistency, Availability, and Partition Tolerance cannot all be guaranteed simultaneously.**
