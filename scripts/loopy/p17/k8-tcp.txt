Here's a thinking process that leads to the explanation of TCP congestion control:

1.  **Deconstruct the Request:**
    *   **Topic:** TCP Congestion Control.
    *   **Key Components to Cover:** Slow Start, Congestion Avoidance, Fast Retransmit, Fast Recovery.
    *   **Level of Detail:** "In detail."

2.  **Initial Knowledge Retrieval & Structuring:**
    *   *What is TCP Congestion Control?* It's a mechanism to prevent network overload (congestion) by adjusting the sender's transmission rate based on network feedback (loss).
    *   *Key Variables:*
        *   `cwnd` (Congestion Window): The number of segments allowed in flight.
        *   `ssthresh` (Slow Start Threshold): A threshold value to switch between algorithms.
        *   `rwnd` (Receive Window): Flow control (not the focus here, but worth noting).
    *   *The Four Phases:*
        1.  **Slow Start:** Exponential growth.
        2.  **Congestion Avoidance:** Linear (additive) growth.
        3.  **Fast Retransmit:** Detecting loss via duplicate ACKs (3 dup ACKs).
        4.  **Fast Recovery:** Recovering from loss without dropping to slow start (Tahoe vs. Reno).

3.  **Drafting the Content - Section by Section:**

    *   **Introduction:**
        *   Define the goal: Prevent network collapse.
        *   Define the mechanism: Feedback loop (Loss = Congestion).
        *   Define the variables (`cwnd`, `ssthresh`).

    *   **Phase 1: Slow Start:**
        *   *Goal:* Probe the network capacity quickly.
        *   *Mechanism:* Start with 1 MSS (Maximum Segment Size). Double the window every RTT (Round Trip Time).
        *   *Visual:* Exponential growth ($1, 2, 4, 8...$).
        *   *Termination:* When `cwnd` > `ssthresh`.

    *   **Phase 2: Congestion Avoidance:**
        *   *Goal:* Once capacity is found, increase more conservatively.
        *   *Mechanism:* Add 1 MSS per RTT (Linear growth).
        *   *Why "Avoidance"?* It avoids the aggressive exponential growth that might cause packet drops.
        *   *Termination:* When a loss occurs (timeout or 3 dup ACKs).

    *   **Phase 3: Fast Retransmit:**
        *   *Problem:* Standard timeout (RTO) is too long (seconds). We need to retransmit faster.
        *   *Mechanism:* Receiver sends Duplicate ACKs (DACKs) when it receives out-of-order segments.
        *   *Trigger:* 3 Duplicate ACKs (3DACKs). This implies the packet was lost, but the connection is still alive and data is flowing.
        *   *Action:* Sender retransmits the missing segment immediately.

    *   **Phase 4: Fast Recovery:**
        *   *Problem:* Standard TCP (Vanilla/Tahoe) drops `cwnd` to 1 MSS (Slow Start) after loss. This is inefficient.
        *   *Mechanism (TCP Reno):*
            *   When 3DACKs arrive, set `ssthresh` to `cwnd` / 2.
            *   Do *not* drop `cwnd` to 1. Keep it high (or slightly reduced).
            *   Increase `cwnd` by 1 MSS for every DACK received (to account for the "extra" data in the pipe).
            *   Once the retransmitted segment is acknowledged, return to Congestion Avoidance.

    *   **Modern Updates (Optional but good for "Detail"):**
        *   Mention TCP Cubic (Linux default) or BBR (Google) briefly to show context, but stick to the prompt's specific request (Reno logic).

4.  **Refining and Formatting:**
    *   Use clear headings.
    *   Use bold text for key terms.
    *   Create a step-by-step logic flow.
    *   *Self-Correction during drafting:* Make sure to distinguish between "Slow Start" (the algorithm) and "Slow Start" (the phase). The phase is called "Slow Start" because it starts small, but the algorithm is exponential.
    *   *Clarification on Fast Recovery:* It's crucial to explain *why*
