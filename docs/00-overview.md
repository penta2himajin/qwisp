# 00 — 基本方針

## 1. 何を作るか

Apple Silicon (MLX) 専用の、**単一モデル特化**ローカル推論エンジン。主ターゲットは Qwen3.6-35B-A3B (MoE)。汎用性を捨て、1 モデルに対して量子化・キャッシュ方策・エージェント連携・検証を深く詰める ―― DS4 の「単一モデル統合に深く最適化」という哲学をそのまま採る。

## 2. DS4 との関係（着想元だが方向は逆）

| | DwarfStar4 (antirez) | Qwisp |
| --- | --- | --- |
| 対象 | DeepSeek V4 Flash (284B) | Qwen3.6-35B-A3B / 27B |
| 動機 | 巨大モデルを高メモリ機(96–128GB)に**ギリ載せる** | スパース性を活かし**より非力な機へ reach** |
| 量子化 | モデル専用の非対称 2/8bit | MoE 向けに再設計が必要（流用不可） |
| 一級市民化 | KV キャッシュをディスクへ | **cold expert をフラッシュへ** |
| プラットフォーム | Metal/CUDA | Apple Silicon / MLX のみ |

DS4 の量子化レシピは「たまたま 2/8 がそのモデルに効いた」事例であり、別アーキの Qwen には**そのまま流用できない**。流用するのは「単一モデルに特化して深く詰める」という方法論の方。

## 3. 中核セオリー：MoE はボトルネックが反転している

LLM の decode は**メモリ帯域律速**。MoE (35B total / 3B active) は：

- **帯域コストは安い**：毎トークン実質 3B 相当しか読まない → 載りさえすれば速い
- **容量コストは高い**：35B 分の重みがどこかに到達可能でなければならない

→ 帯域はコードで超えられない物理上限だが、**容量は low-bit 量子化・KV 量子化・expert streaming で攻められる**。よって「MoE を載せる工夫」に投資する価値がある。これは Apple の **AFM 3 Core Advanced**（20B をフラッシュ常駐、1–4B だけ DRAM ロード）と同じ構図で、本プロジェクトはその自前 OSS 版を狙う。詳細は `01-research-notes.md`。

## 4. 過拘束の認識（重要）

「**最弱 M1 (8GB)** × **Qwen3.6 27B/35B** × **十分高性能**」は三立しない。

- Qwen3.6-35B-A3B ≈ Q4 で 21GB、27B ≈ Q4 で 17GB。8GB には**載らない**。16GB でも 27B を IQ3 まで落として辛うじて、35B-A3B は不可。
- 決定打：Apple 自身が AFM 3 Core Advanced を「最も高性能な Apple silicon」に限定＝最新チップのみ。**フルスタックを握る Apple ですら弱いチップには載せない**。

→ **フロアは引き上げる**。現実線は 24GB 級ユニファイドメモリ。最弱 M1/8GB は明示的に圏外（or「動きはする」ティア）とする。最終決定は実測（`02-roadmap.md` Step 3）後。

## 5. 設計上の基準（暫定）

- ベースは **MLX**（Apple 上で llama.cpp 比 23–41% 速い、Qwen3.6 の Gated DeltaNet 遅延問題が出ない）
- レバー：MoE expert streaming + 混合精度 expert + KV 量子化 + **Multi-Token Prediction / 投機デコード**（帯域律速機ほど重み読み償却が効く）
- **「十分高性能」を必ず 4 つ組で固定する**：`{モデル+量子化, 最低 tok/s, context 長, 正確なフロア機種+RAM}`。未定義のままだとプロジェクトがドリフトする。

## 6. スコープ外（少なくとも初期）

- 非 Apple プラットフォーム
- IFP 等のモデル再学習を要する手法（やるなら別フェーズ。`01` C 節参照）
- 27B dense の速度最適化（dense は帯域で詰むため、MoE を主軸に置く判断）
